# 日本語フォントで全角、コンソールで半角幅扱いになる文字の問題

Unicode は各文字に「東アジア幅 (East Asian Width)」プロパティ ([UAX #11](https://www.unicode.org/reports/tr11/)) を定義しています。そのうち "Ambiguous" (曖昧幅) に分類される文字は、CJK (中日韓) 環境では全角 2 セル幅、西洋環境では半角 1 セル幅として扱います。日本語フォントはこれらに全角グリフを割り当てることが多い一方で、コンソール (端末エミュレーター) は半角 1 セルとして描画するため、文字幅の不一致が生じます。

## Unicode East Asian Width の分類

UAX #11 は各文字の表示幅を 6 種類のプロパティ値で定義しています。

| プロパティ値 | 略称 | 幅の扱い |
|---|---|---|
| Fullwidth | F | 常に全角 (2 セル) |
| Wide | W | CJK 環境で全角 |
| Halfwidth | H | 常に半角 (1 セル) |
| Narrow | Na | 西洋環境で半角 |
| Neutral | N | 文脈依存 (既定は半角) |
| Ambiguous | A | 文脈依存 (CJK で全角、西洋で半角) |

問題になるのは Ambiguous (A) の文字群で、文脈が確定できない場合は半角として扱うことが規格上の既定とされています。

## 影響を受ける主な文字範囲

以下のコード ポイント範囲が Ambiguous に分類される代表例で、日本語フォントでは全角グリフが割り当てられていることが多いです。

| コード ポイント範囲 | 文字の種類 | 例 |
|---|---|---|
| U+00B0-U+00FF | ラテン補助 (記号類) | `°` `±` `×` `÷` |
| U+0370-U+03FF | ギリシャ文字 | `α` `β` `γ` `Ω` |
| U+0400-U+04FF | キリル文字 | `А` `Б` `я` |
| U+2013-U+203B | 一般句読点 | `–` `—` `…` `※` |
| U+2018-U+201D | 引用符 | `'` `'` `"` `"` |
| U+2190-U+21FF | 矢印 | `←` `→` `↑` `↓` |
| U+2500-U+257F | 罫線文字 (Box Drawing) | `─` `│` `┌` `┘` |
| U+25A0-U+25FF | 幾何図形 | `■` `□` `●` `○` `▲` |
| U+2600-U+26FF | その他記号 | `★` `☆` `♪` |

罫線文字と幾何図形はテキスト UI やターミナルの表組みで頻繁に使われるため、実害が大きいです。

## 日本語フォントの挙動

以下の日本語フォントは Ambiguous 文字に対して全角 (2 セル相当) グリフを収録しています。

- MS ゴシック / MS 明朝
- メイリオ (Meiryo)
- 游ゴシック / 游明朝 (Yu Gothic / Yu Mincho)
- BIZ UD ゴシック / UD 明朝
- Noto Sans/Serif CJK JP
- IPA フォント各種

これらのフォントをターミナルに設定すると、グリフが 2 セル幅で描画されながらカーソルは 1 セルしか進まないため、後続の文字と重なります。

## コンソール側の挙動

端末エミュレーターは `wcwidth()` / `wcswidth()` 相当のロジックで文字幅を判定します。Ambiguous 文字の扱いは実装によって異なります。

| 環境 | Ambiguous のデフォルト扱い | 設定変更 |
|---|---|---|
| Windows Terminal | 半角 (1 セル) | 未対応 ([issue #10844](https://github.com/microsoft/terminal/issues/10844)) |
| VS Code 統合ターミナル | 半角 (1 セル) | 未対応 |
| GNOME Terminal / Konsole | 半角 (1 セル) | 対応 (要確認) |
| iTerm2 (macOS) | 設定可 | 対応 |
| Alacritty | 設定可 | [PR #1049](https://github.com/alacritty/alacritty/pull/1049) で対応 |
| mlterm | 設定可 | 対応 |

## 発生する問題

コンソール出力で表組みや TUI を描画するとき、Ambiguous 文字を全角グリフで表示しながらカーソル位置計算が半角 1 セル分しか進まないと、以下の崩れが起きます。

- 罫線が次の文字に重なってテーブル枠が壊れる
- `printf` の列幅指定がずれて右端が揃わない
- `tmux` や `vim` がリフレッシュするたびに不要な改行を挿入する

## 対処方針

問題の解決は、コンソール側・フォント側・アプリ側の 3 層それぞれで取り組めます。

コンソール側では、端末エミュレーターの "Ambiguous character width" 設定を "Wide (2 cell)" にそろえます。ただし Windows Terminal は現時点でこの設定を持ちません。

フォント側では、Ambiguous 文字に半角グリフを持つ専用フォント (例: [Illusion](https://qiita.com/tomonic-x/articles/601343ec3e5f6bd59c0d)) を使うことで、フォント描画とカーソル位置を一致させられます。

アプリ側では、コンソール出力に Ambiguous 文字を使わないことが最も移植性の高い対策となります。特に表組みや罫線には ASCII 記号 (`|` `-` `+`) を使います。

Markdown ドキュメントを日本語環境で編集する場合も同じ注意が必要です。ファイル ツリーや階層構造を表現するために罫線文字 (`└` `├` `─` など) を使うと、日本語フォントが全角グリフで描画する一方でプレビューやコンソールは半角幅として処理するため、インデントがずれます。ツリー構造や階層の表現には ASCII 記号 (`+` `-` `|`) を使うことで、フォントや環境による幅のばらつきを回避できます。

## まとめ

| 層 | 対策 | 備考 |
|---|---|---|
| アプリ | Ambiguous 文字を避け ASCII 記号を使う | 最も確実 |
| フォント | 半角グリフを持つ端末専用フォントを使う | Illusion 等 |
| コンソール | Ambiguous width 設定を Wide に変更 | 対応端末のみ |

フォントとコンソールの組み合わせによって挙動が変わるため、アプリ側で Ambiguous 文字に依存しない設計にしておくことが、クロスプラットフォームでの安定した表示に最も効果的です。
