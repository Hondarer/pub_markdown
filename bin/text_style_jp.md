# text_style_jp - 日本語テキスト スタイ リング コマンド

## 概要

`text_style_jp` は、Microsoft 日本語スタイル ガイドに基づき、日本語技術文書とソース コード コメントの表記を統一するコマンドです。全角・半角の変換、NBSP (U+00A0) の半角スペース化、単語間スペースの挿入・削除を自動で行い、一貫性と可読性を向上させます。

Markdown に加えて、C/C++、C#、Python、Shell、Makefile のソース コード コメントも扱えます。  
また、全モード共通で NBSP (U+00A0) を半角スペースへ置換します。さらに、連続する複数の空行を 1 つの空行に圧縮し、出力末尾には必ず改行を付与します。

## 背景

### Microsoft 日本語スタイル ガイドとは

Microsoft が自社製品のローカライズ用に策定した日本語表記ガイドラインです。Microsoft Language Portal からダウンロード可能で、IT 業界で広く参照されています。

参照元:

- [Microsoft Localization Style Guides](https://learn.microsoft.com/en-us/globalization/reference/microsoft-style-guides)
- [Japanese Style Guide (PDF)](https://download.microsoft.com/download/a/8/2/a822a118-18d4-4429-b857-1b65ab388315/jpn-jpn-StyleGuide.pdf)

### 類似のスタイル ガイド

| ガイド | カタカナ複合語の区切り | 特徴 |
|--------|------------------------|------|
| Microsoft | 半角スペース | IT 業界標準、製品ローカライズ向け |
| IBM | 中黒 (・) | 伝統的な表記 |
| JTF (日本翻訳連盟) | 中黒 or 半角スペース | 翻訳業界標準、CC BY ライセンス |
| Apple | 区切りなし | シンプルさ重視 |

本コマンドは Microsoft スタイルを採用しています。

## 実装ルール

### 文字種の変換

#### 全角英数字・括弧類 → 半角

全角の ASCII 英数字 (A-Z, a-z, 0-9) および括弧類 (()[]{}) は半角に変換します。

| 変換前 | 変換後 |
|--------|--------|
| `ＮＯＴＥ` | `NOTE` |
| `１２３` | `123` |
| `Ｗｉｎｄｏｗｓ` | `Windows` |
| `（タイトル）` | `(タイトル)` |
| `［新規］` | `[新規]` |
| `｛key｝` | `{key}` |

#### 全角疑問符・感嘆符 → 半角

全角の疑問符 `？` および感嘆符 `！` は半角に変換します。

| 変換前 | 変換後 |
|--------|--------|
| `本当ですか？` | `本当ですか?` |
| `すごい！` | `すごい!` |

ルール ID: `fullwidth-bang`

#### 半角カタカナ → 全角

半角カタカナは全角に変換します。濁点・半濁点の結合も処理します。

| 変換前 | 変換後 |
|--------|--------|
| `ｶﾀｶﾅ` | `カタカナ` |
| `ｶﾞｷﾞｸﾞ` | `ガギグ` |
| `ﾊﾟｿｺﾝ` | `パソコン` |

### スペース挿入ルール

#### 全角文字と半角英数字の間

全角文字 (漢字、ひらがな、全角カタカナ) と半角英数字の間に半角スペースを挿入します。

| 変換前 | 変換後 | 説明 |
|--------|--------|------|
| `第3章` | `第 3 章` | 漢字と数字 |
| `Windows10を起動` | `Windows10 を起動` | 英数字とひらがな |
| `ABCあいう` | `ABC あいう` | 英字とひらがな |
| `データAを参照` | `データ A を参照` | カタカナと英字 |

このルールは Microsoft スタイル ガイドの中核であり、プロポーショナル フォント環境でも視覚的な区切りを明確にする効果があります。

#### カタカナ複合語

英語で複数単語から成るカタカナ語は半角スペースで区切ります。

| 変換前 | 変換後 |
|--------|--------|
| `ダイアログボックス` | `ダイアログ ボックス` |
| `メニューバー` | `メニュー バー` |
| `コントロールパネル` | `コントロール パネル` |

分割と保護の判断は、Microsoft Terminology Collection (JAPANESE.tbx) から自動生成した辞書を主軸に行います。

- tbx に **カタカナ語として登録された語** (単独語と複合語の構成要素。例: `ワークスペース` `サブディレクトリ` `クロスプラットフォーム` `トラブルシューティング` `インライン`) は `no_space` リストとして自動登録され、分割対象から除外されます。
- tbx の **スペース区切りカタカナ複合語** (例: `ダイアログ ボックス` `ファイル マネージャー`) は `add_space` ペアとして自動登録され、連結形が標準形へ分割されます。
- tbx 由来の `no_space` に登録されていない残りのカタカナ列のみ、SudachiPy (形態素解析、モード B) によりフォールバックで分割されます。

SudachiPy が形態素として認識できない語や、`replace` 適用後に初めて分割できる語については `add_space` エントリで補います。詳細は「辞書ファイル」セクションを参照してください。

### スペース削除ルール

#### 句読点の前

句読点、コロン、セミコロン、疑問符、感嘆符の直前のスペースを削除します。  
ただし、相対パスの `./` や `.\` の直前スペースは保持します。

| 変換前 | 変換後 |
|--------|--------|
| `クリックして 、` | `クリックして、` |
| `閉じます 。` | `閉じます。` |
| `更新しますか ?` | `更新しますか?` |
| `警告 !` | `警告!` |
| `フォント :` | `フォント:` |
| `= と :=` | `= と :=` |
| `先頭の ./ は` | `先頭の ./ は` |

#### 括弧の内側

開き括弧の直後、閉じ括弧の直前のスペースを削除します。

| 変換前 | 変換後 | 括弧の種類 |
|--------|--------|-----------|
| `( タイトル )` | `(タイトル)` | 丸括弧 |
| `[ 新規 ]` | `[新規]` | 角括弧 |
| `「 test 」` | `「test」` | 鉤括弧 |

#### スペース不要の単位記号

度 (°)、パーセント (%) の前のスペースを削除します。

| 変換前 | 変換後 | 備考 |
|--------|--------|------|
| `45 °` | `45°` | 角度 |
| `50 %` | `50%` | パーセント |
| `50 ％` | `50%` | 全角パーセント→半角変換後 |

また、写真・映写関連の単位 `mm` もスペース不要とします。

| 変換前 | 変換後 |
|--------|--------|
| `35 mm` | `35mm` |

他の単位記号 (kg, m, km など) は数値との間に半角スペースを入れる規則がありますが、本コマンドでは自動挿入は行いません。

### スペース挿入の補完ルール

#### 疑問符・感嘆符の後

疑問符・感嘆符 (`?` `!`) の直後に半角英数字または日本語が続く場合、スペースを挿入します。  
ただし直後が全角の句読点・閉じ括弧 (`。、！？）」` 等) の場合はスペースを挿入しません。

| 変換前 | 変換後 |
|--------|--------|
| `保存しますか?Excelを` | `保存しますか? Excel を` |
| `できますか?はい` | `できますか? はい` |
| `すごい!本当` | `すごい! 本当` |
| `質問?」と言った` | `質問?」と言った` (変換なし) |

全角 `？` `！` は `fullwidth-bang` ルールで先に半角化されるため、元から全角でも同じ結果になります。

ルール ID: `space-after-punctuation`

#### ページ番号形式 + 括弧

`数字/数字` の直後に括弧が続く場合、スペースを挿入します。

| 変換前 | 変換後 |
|--------|--------|
| `10/13(ページ)` | `10/13 (ページ)` |

#### 補足括弧

英数字や `/` を含む識別子、代入演算子、強調の直後、`○` や `✅` などの記号の直後に補足括弧が続く場合は、括弧の前にスペースを挿入します。  
ただし、強調の開始記号と括弧の間にはスペースを挿入しません。

| 変換前 | 変換後 |
|--------|--------|
| `**強調**(補足)` | `**強調** (補足)` |
| `+=(追記)` | `+= (追記)` |
| `UDP/IPv4(unicast / multicast)` | `UDP/IPv4 (unicast / multicast)` |
| `○(双方向)` | `○ (双方向)` |
| `✅(複数 path からの重複排除)` | `✅ (複数 path からの重複排除)` |
| `**(Linux のみ)**` | `**(Linux のみ)**` |
| `O(1)` | `O(1)` |
| `全角コロン：サンプル` | `全角コロン:サンプル` |
| `リオーダーバッファ` | `リオーダー バッファー` |
| `リオーダーバッファタイムアウト` | `リオーダー バッファー タイムアウト` |
| `シグナルハンドラ` | `シグナル ハンドラー` |
| `フィルタファイル` | `フィルター ファイル` |
| `フィルタルール` | `フィルター ルール` |
| `フィルタ・セッション` | `フィルター・セッション` |

### Markdown 固有の処理

#### コード ブロックの保護

コード ブロック (` ` `` ` で囲まれた範囲) 内はスタイ リング対象外とします。

```text
# 変換対象
第3章 概要

# 変換対象外
```python
print("第3章")
```
```

#### インライン コードの保護

バッククォート (`` ` ``) で囲まれたインライン コードはスタイ リング対象外とします。

| 入力 | 出力 |
|------|------|
| `` `第3章`を参照 `` | `` `第3章` を参照`` |
| `` ` ```text ` `` | `` ` ```text ` `` |
| `Linux Kernel は**Sphinx**を採用` | `Linux Kernel は **Sphinx** を採用` |
| `` reStructuredText 形式で`make htmldocs` または`make pdfdocs` `` | `` reStructuredText 形式で `make htmldocs` または `make pdfdocs` `` |
| `## @brief { brief description }` | `## @brief { brief description }` |
| `## @param[<dir>] <parameter-name> { parameter description }` | `## @param[<dir>] <parameter-name> { parameter description }` |
| `## @f[ ~ @f], @f$ ~ @f$` | `## @f[ ~ @f], @f$ ~ @f$` |
| `##?=(条件付き代入)` | `## ?= (条件付き代入)` |
| `**リンク補完の配置先：**` <br>`` `/INCLUDE` pragma は `mock_com_util.h` ヘッダーに記述します。`` | `**リンク補完の配置先:**` <br><br>`` `/INCLUDE` pragma は `mock_com_util.h` ヘッダーに記述します。`` |

#### URL の保護

`https://` または `http://` で始まる URL はスタイ リング対象外とします。  
URL 末尾が日本語句読点 (`。`、`、`、`」` など) の場合、句読点は URL の一部とみなしません。

| 入力 | 出力 |
|------|------|
| `詳しくは https://example.com/日本語abc を参照` | `詳しくは https://example.com/日本語abc を参照` |
| `[text](https://example.com/日本語パス)` | `[text](https://example.com/日本語パス)` |
| `参照先 https://example.com/doc。次の章` | `参照先 https://example.com/doc。次の章` |

#### コード フェンス前後の空行

Markdownlint の MD031 相当として、コード フェンスの前後に本文が続く場合は空行を 1 行挿入します。  
文書の先頭または末尾にあるコード フェンスには余分な空行を追加しません。

| 入力 | 出力 |
|------|------|
| `例:` <br>```makefile<br>LIBS += mock_calcbase mock_libc<br>``` | `例:` <br><br>```makefile<br>LIBS += mock_calcbase mock_libc<br>``` |
| `例:` <br>```makefile<br>LIBS += mock_calcbase mock_libc<br>``` <br>`補足` | `例:` <br><br>```makefile<br>LIBS += mock_calcbase mock_libc<br>``` <br><br>`補足` |

#### リスト ネスト字下げの正規化

Markdown のリスト マーカーのネスト字下げを **depth × 4 スペース** に正規化します。  
トップレベル (depth 0) の項目はそのままにし、ネスト項目 (depth 1 以上) のみが対象です。  
マーカー行に加え、その項目に属する継続テキスト行・配下のフェンス付きコード ブロックも同一 delta で追従します。

| depth | スペース数 | 例 |
|-------|----------|-----|
| 0 | 変更なし | `- 親` |
| 1 | 4 | `    - 子` |
| 2 | 8 | `        - 孫` |

| 入力 | 出力 | 説明 |
|------|------|------|
| `- 親` <br>`  - 子` | `- 親` <br>`    - 子` | depth-1 を 4 スペースへ |
| `- 親` <br>`  - 子` <br>`    - 孫` | `- 親` <br>`    - 子` <br>`        - 孫` | 多段ネストも追従 |
| `1. 親` <br>`   - 子` | `1. 親` <br>`    - 子` | 順序付きと記号の混在 |
| `- 親` <br>`    - 子` | `- 親` <br>`    - 子` | すでに 4 スペースは変更なし |

ルール ID: `list-indent`

#### 見出し・本文の直後に続く top-level list の前空行

Markdownlint の MD032 相当として、見出しまたは本文の直後に top-level list が続く場合は、list の前に空行を 1 行挿入します。

| 入力 | 出力 |
|------|------|
| `### 非侵襲的` <br>`- テスト対象の C コードを一切変更する必要がない` | `### 非侵襲的` <br><br>`- テスト対象の C コードを一切変更する必要がない` |

#### 行末 trailing spaces の正規化

Markdown では行末の半角スペース 2 つ以上がソフト改行 (line break) として機能しますが、  
次行が空行または文末 (EOF) の場合は段落の区切りが明確なためソフト改行は不要となります。  
このため、次行が非空行の本文行では行末を半角スペース 2 個に正規化し、  
それ以外の行末 trailing spaces は除去します。

| 入力 | 出力 | 説明 |
|------|------|------|
| `行1` + 次行が `行2` | `行1  ` | ソフト改行として半角スペース 2 個を付与 |
| `行1   ` + 次行が `行2` | `行1  ` | trailing spaces を半角スペース 2 個に正規化 |
| `単独の段落  ` + 次行が空行 | `単独の段落` | trailing spaces を除去 |
| `行1  ` + 文末 (EOF) | `行1` | trailing spaces を除去 |

コード ブロック内の行は対象外とします。

## 辞書ファイル

スペース挿入ルールをプロジェクト・ユーザーごとにカスタマイズできます。

### 置き場所

以下の 3 つのディレクトリを検索対象とし、各ディレクトリの `*.json` をすべて収集してから **ファイル名昇順で統一ソートして読み込みます**。

1. `~/.text_style_jp/*.json` — ログイン ユーザーのホーム ディレクトリ設定
2. `<docsfw>/.text_style_jp/*.json` — docsfw サブモジュール共通の設定
3. `<カレントディレクトリ>/.text_style_jp/*.json` — カレント ディレクトリの設定

パスが重複する場合は自動的に除去されます (例: `<docsfw>` がカレント ディレクトリと一致する場合)。  
同名ファイルが複数のディレクトリに存在する場合のみ、上記の順 (ホーム → docsfw → カレント) でタイ ブレークします。

ファイルはソート順で読み込まれ、**後から読み込まれた定義が優先されます** (後勝ち)。したがって、`NN_name.json` 形式の 2 桁数字プレフィックスを使えば、どのディレクトリに配置した辞書でもファイル名で優先順位を一元的に管理できます。

例:

- `10_microsoft.json` — TBX から自動生成したベース辞書 (no_space、replace 順方向/逆方向、add_space)
- `20_adverb-kana.json` — 一般的なかな化ルール
- `30_testfw-keywords.json` — 製品固有の除外ルール
- `90_local-katakana.json` — TBX 未収録の社内独自カタカナ語、tbx との衝突を上書きするオーバーライド

ホーム ディレクトリに置いた `99_user-override.json` は、docsfw 共通の `10_microsoft.json` より高優先になります。

### フォーマット

辞書ファイルは `.json` 拡張子のまま、JSONC 形式の `//` コメント、`/* ... */` コメント、末尾カンマを使用できます。

```json
{
  // スペースを挿入しない語
  "no_space": [
    "Pre-Assert確認",
    "CI/CD環境",
  ],
  "add_space": [
    {"from": "タスクマネージャー", "to": "タスク マネージャー"},
    {"from": "ダイアログボックス", "to": "ダイアログ ボックス"},
  ],
  "replace": [
    {"from": "コンピュータ", "to": "コンピューター"},
    {"from": "プリンタ", "to": "プリンター"},
  ],
}
```

| キー | 説明 |
|------|------|
| `no_space` | カタカナ語の許可リスト。全角/半角境界へのスペース挿入、SudachiPy による過剰分割、`replace` の誤適用を抑制します。カタカナ列のスペース挿入誤記 (例: `サブ モジュール`) も連結形が登録されていれば自動的に補正されます |
| `add_space` | スペースを挿入する変換ペアのリスト。SudachiPy が対応できない語 (`replace` 適用後に初めて分割できる語など) を補います |
| `replace` | 汎用文字列置換の変換ペアのリスト (長音記号の付与・削除、かな化など) |

`_comment` など未知のキーは無視されます (コメント代わりに使用可)。説明だけを書く場合は JSONC コメントも使用できます。

#### VS Code での定義

VS Code の files.associations は glob 風のパターンを指定できるので、.text_style_jp 配下の .json を JSONC として扱えます。

例:

```json
"files.associations": {
    "**/.text_style_jp/*.json": "jsonc",
}
```

`**/.text_style_jp/*.json` なら、ホーム・docsfw・カレント ディレクトリなど、どの階層の .text_style_jp 配下でも .json が JSONC として扱われます。

#### no_space と add_space の衝突時の優先順位

同一語が `no_space` と `add_space` の両方に登録される場合、以下のルールで最終分類を決定します。

- **同一ファイル内**: `add_space` が `no_space` に勝ちます
- **異なるファイル間**: ファイル名ソート順で後ファイルが勝ちます (`no_space` / `add_space` のどちらの分類かを上書きします)

例: `10_microsoft.json` は「ライフサイクル」を `no_space` と `add_space` の両方に持ちます (tbx の揺らぎ)。同一ファイル内なので `add_space` 側が採択され、デフォルトでは「ライフサイクル」に変換されます。  
`90_local-katakana.json` の `no_space` に「ライフサイクル」を追加すれば、後発の `90_` がファイル名順で勝ち、結合形「ライフサイクル」が維持されます。

`10_microsoft.json` は `tbx_to_dict.py` により JAPANESE.tbx から自動生成され、次の内容を含みます。

- **`no_space`**: tbx に登録されたカタカナ語 (単独語と複合語の構成要素) のうち長さ 3 以上。これにより `ワークスペース` `サブディレクトリ` `クロスプラットフォーム` `トラブルシューティング` `インライン` などが過剰分割されません
- **`replace` (順方向 ー 付与)**: tbx の `ー` で終わる単独語から、その `ー` を除いた形を `from` とするペア。例: `サーバ → サーバー`、`ユーザ → ユーザー`
- **`replace` (逆方向 ー 削除)**: tbx 単独語 `w` のうち、`w + ー` が tbx に登録されていない場合に `w + ー → w` のペアを生成します。例: `カテゴリー → カテゴリ`、`メモリー → メモリ`、`プロパティー → プロパティ`。語末音が tbx 内の `ー` 付与パターンと一致する語に限定して生成し、`コンピューター` のように両方が tbx 標準形として併存する語は除外します
- **`add_space`**: tbx のスペース区切り複合語から、スペースを除いた連結形を `from` とするペア。例: `ファイルマネージャー → ファイル マネージャー`

`tbx_to_dict.py` を再実行することで JAPANESE.tbx 更新時に辞書を再生成できます。

`90_local-katakana.json` には主に次の 2 種類の登録を行います。

- **TBX 未収録の社内独自カタカナ語**: `no_space` に追加するとスペース挿入誤記 (例: `サブ モジュール`) も自動的に結合形へ補正されます
- **TBX の揺らぎをオーバーライドしたい語**: TBX に `no_space` と `add_space` の両方に登録されて `add_space` 側が採択されている語を `no_space` に追加すると、`90_` のファイル名が `10_` より後順になるため結合形を維持できます (例: `ライフサイクル` を追加すると「ライフサイクル」への分割を抑制できます)

辞書の処理順序は次のとおりです。

1. `replace` を Sudachi 分割前に先行適用 — 逆方向 `カテゴリー → カテゴリ` や順方向 `スライドショ → スライドショー` を、Sudachi B 分割や `no_space` 保護で分断される前に処理します
2. `no_space` プレースホルダー化 (長い順マッチ)
3. SudachiPy モード B でカタカナ列を分割
4. `no_space` 復元 (カタカナ境界に自動スペース補完、ただし `ー` と `・` は境界扱いしません)
5. `replace` を再適用 (Sudachi 分割後に初めて連続する語に対応)
6. `add_space` 適用 (連結形を Microsoft 標準のスペース区切りへ)

### 注意事項

- インライン コード (`` ` `` で囲まれた範囲) 内は変換対象外です
- コード ブロック (` ` `` ` で囲まれた範囲) 内は変換対象外です
- HTML コメント ブロック (`<!-- ... -->`) と、`<!--ja:--> ... <!--:ja-->` / `<!--en: ... :en-->` のようなコメント タグ領域は変換対象外です
- `no_space` の長い単語から順にプレースホルダー化するため、部分一致による誤置換は発生しません
- 全カタカナ同士の `replace` は、前後が非カタカナまたは行境界のときだけ一致します
- `no_space` に登録した語は `replace` の直前に再保護されるため、長音付与などの置換対象になりません
- 読み込みエラー時はその辞書ファイルを無視して処理を継続します
- Markdown 見出しの番号除去は `## 1. タイトル` のような見出し番号に限定し、`### 2 回以上の制限` や `#### 1 階層下まで指定` のような数量表現・内容上の数値は保持します
- SudachiPy が未インストールの場合は初回実行時に自動インストールを試みます (`--user` スコープ)。インストールできない場合は `add_space` 辞書のみで動作します

## 使用方法

### コマンド ライン

```bash
# Markdown ファイルを処理
python text_style_jp.py input.md --mode markdown -o output.md

# 拡張子から自動判定してコメントを処理
python text_style_jp.py input.c --mode auto -o output.c

# 標準入出力
cat input.md | python text_style_jp.py --mode markdown > output.md

# モードを明示して処理
python text_style_jp.py input.cs --mode csharp -o output.cs

# 未対応拡張子など、NBSP 正規化と空行圧縮、末尾改行付与を行う
python text_style_jp.py input.txt --mode text -o output.txt

# チェックのみ (CI 向け)
python text_style_jp.py --mode auto --check input.py

# 変更内容の確認 (dry-run)
python text_style_jp.py --dry-run input.md

# テスト実行
python text_style_jp.py --test
```

### dry-run モード

`--dry-run` を指定すると、ファイルを変更せずに「どこが・どのルールで・どの辞書によって変更されるか」を textlint 風フォーマットで表示します。変更が 1 件以上あれば終了コード 1 を返します。NBSP の検出も対象で、Markdown のコード フェンスやソース中の文字列リテラルを含む入力全体を確認します。NBSP は表示上 `\u00A0` として出力されます。

```
file.md
  3:5   "第3章" → "第 3 章"   fullwidth-halfwidth-space
  5:10  "サーバ" → "サーバー"  dict-replace (10_microsoft.json)
  7:1   " 1." → ""            heading-number

  3 problems found
```

出力フォーマット:

```
  {行}:{列}  "{変更前}" → "{変更後}"  {ルール ID}[ ({辞書ファイル名})]
```

### Python モジュールとして

```python
from text_style_jp import style_markdown, style_source_comments, style_prose

# Markdown 全体を処理 (本文整形ではコード ブロック保護あり、NBSP は全文対象)
styled = style_markdown(text)

# C ソースのコメントを整形 (NBSP は全文対象)
styled = style_source_comments(source_text, "c")

# 単純なテキストを処理
styled = style_prose("第3章")  # "第 3 章"
```

## 処理順序

スタイ リングは以下の順序で適用されます。

1. 入力全体の NBSP (U+00A0) を半角スペースへ置換
2. コード フェンス・インライン コード・URL を保護 (変換対象外化)
3. カタカナ列に挟まれた半角スペースを除去 (連結形が `no_space` に登録されている場合のみ)
4. `replace` を先行適用 (Sudachi 分割や `no_space` 保護で連続性が失われる前に逆方向 ー 削除や順方向 ー 付与を処理)
5. `no_space` 語をプレースホルダー化
6. 全角英数字・括弧類を半角に変換
7. 半角カタカナを全角に変換
8. 全角スペースを半角に変換、連続スペースを 1 つに正規化
9. 全角文字と半角英数字の間にスペースを挿入
10. 句読点前のスペースを削除
11. 括弧内側のスペースを削除
12. 単位記号 (°, %, mm) 前のスペースを削除
13. 疑問符・感嘆符後に英数字が続く場合スペースを挿入
14. ページ番号形式後の括弧前にスペースを挿入
15. スペースの正規化
16. SudachiPy (モード B) でカタカナ連続部分を形態素分割
17. `no_space` 語を復元 (カタカナ境界スペースを自動補完、`ー` と `・` は境界扱いしない)
18. `replace` を適用
19. `no_space` 語を再復元 (カタカナ境界スペースを自動補完)
20. `add_space` を適用
21. URL・保護パターンを復元
22. ソフト改行と見なせない行末の不要な trailing spaces を除去 (Markdown 固有)
23. 連続する複数の空行を 1 つの空行に圧縮し、出力末尾に改行を付与 (全モード共通)

## 対応モード

| モード | 主な対象 |
|--------|----------|
| `markdown` | Markdown 本文。コード フェンスとインライン コードは保護します |
| `c`, `cpp` | `//`、`/* */`、Doxygen コメント |
| `csharp` | `//`、`/* */`、`///` XML ドキュメント コメント |
| `python` | `#` コメント |
| `shell` | `#` コメント |
| `make` | `#` コメント |
| `text` | 空行圧縮と末尾改行付与のみ |

`--mode auto` を指定した場合は、ファイル名と拡張子からモードを推定します。未対応拡張子は `text` として扱います。

## 制限事項

### 未実装の機能

以下の Microsoft スタイル ガイドの規則は本コマンドでは未実装です。

| 項目 | 例 | 理由 |
|------|-----|------|
| 数字と単位間のスペース | 3kg → 3 kg | 単位の判定が困難 |
| 数値の漢数字変換 | 一つ → 1 つ | 文脈依存 |
